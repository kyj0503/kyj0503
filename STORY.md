### 2.1. Personal Project
#### 2.1.1. 라고할때살걸
[웹사이트](https://backtest.yeonjae.kr/), [Github](https://github.com/kyj0503/backtest)

**기간:** 2025. 05. ~ 2025. 11.

과거 주가 데이터에 적립식 매수(DCA), 리밸런싱 등 다양한 투자 전략을 대입해 수익률을 시뮬레이션하는 고성능 백테스팅 서비스입니다.

강남대학교 컴퓨터공학부 졸업작품전시회 **장려상** 수상

* **Architecture & Design Pattern**
    * **Custom Dependency Injection:** 외부 프레임워크에 의존하지 않고, Python의 동적 타이핑 특성을 활용한 **DI Container를 직접 구현**하여 서비스 간 결합도를 낮추고 테스트 용이성을 확보했습니다. (Singleton Registry 패턴 적용)
    * **Strategy & Factory Pattern:** 다양한 보조지표(RSI, Bollinger Bands 등)와 매매 전략을 **Strategy 패턴**으로 추상화하여, 기존 코드 수정 없이 새로운 전략을 플러그인처럼 확장할 수 있는 유연한 구조를 설계했습니다.
    * **FastAPI & Async:** I/O 바운드 작업(DB, 외부 API)과 CPU 바운드 작업(백테스팅 연산)을 효율적으로 처리하기 위해 비동기 아키텍처를 도입했습니다.

* **Performance & Optimization**
    * **Vectorization (Pandas/Numpy):** 기존 $O(N^2)$ 복잡도를 가지던 일별 수익률 계산 로직을, **벡터 연산**을 통해 사전 정렬 및 일괄 처리하는 방식으로 재설계하여 시뮬레이션 속도를 획기적으로 단축했습니다. [블로그 포스팅](https://blog.naver.com/kanden9999/224105484295)
    * **Async/Sync Boundary Handling:** Pandas와 같은 동기(Blocking) 라이브러리가 비동기 이벤트 루프를 차단하지 않도록 `asyncio.to_thread`를 활용해 스레드 풀 격리 전략을 적용했습니다. [블로그 포스팅](https://blog.naver.com/kanden9999/224084582339)
    * **Smart Caching & Downsampling:** 데이터 조회 빈도에 따른 계층적 캐싱(Local -> DB -> API)과, 프론트엔드 차트 렌더링 부하를 줄이기 위한 시계열 데이터 다운샘플링 알고리즘을 구현했습니다.

* **Data Consistency & Reliability**
    * **Nth Weekday Algorithm:** "매월 두 번째 수요일"과 같은 복잡한 리밸런싱 주기를 정확히 계산하기 위해 `dateutil`에 의존하지 않고 캘린더 알고리즘을 직접 구현하여 비즈니스 로직의 정확도를 높였습니다.
    * **Robust Database Interaction:** 트랜잭션 데드락 발생 시 지수 백오프를 적용한 **자동 재시도 메커니즘**을 구현하여 시스템 안정성을 강화했습니다.
    * **Connection Pool Management:** 외부 API 연동 시 간헐적인 커넥션 유실 문제를 해결하기 위해, 데이터 저장 트랜잭션 직후 커넥션을 명시적으로 재수립하는 방어 로직을 적용했습니다. [블로그 포스팅](https://blog.naver.com/kanden9999/224086753604)

**역할:** 백엔드 팀장 (FE 3, BE 4)

#### 2.2.2. Jandi Web IDE
[웹사이트](https://jandiide.netlify.app/), [Github](https://github.com/kyj0503/jandi_ide_backend)

**기간:** 2025. 04.

**역할:** 백엔드 팀장 (FE 2, BE 3)

실시간 코딩 테스트 연습 및 개발자 협업을 위한 웹 IDE 플랫폼입니다. 불안정한 사용자 코드를 안전하게 실행하기 위한 프로세스 격리 기술과 대용량 실시간 메시지 처리를 위한 분산 아키텍처 설계에 주력했습니다.

* **안정적인 원격 코드 실행(RCE) 엔진 개발**
    * **문제:** 사용자가 제출한 코드가 무한 루프에 빠지거나 과도한 메모리를 점유할 경우, WAS(Tomcat) 전체가 마비되거나 서버가 다운되는 심각한 보안 및 가용성 문제가 있었습니다.
    * **해결:**
        1.  **프로세스 격리:** `ProcessBuilder`를 사용하여 사용자 코드를 메인 서버 스레드와 분리된 별도의 OS 프로세스에서 실행하도록 격리했습니다.
        2.  **리소스 제한:** Java(`-Xmx`), Python, C++ 등 언어별 컴파일/실행 명령어에 메모리 제한 옵션을 적용하고, `waitFor(time, unit)` 메서드를 활용하여 설정된 시간을 초과하면 프로세스를 강제 종료(`destroyForcibly`)시키는 타임아웃 로직을 구현했습니다.
        3.  **파일 시스템 관리:** `CompilerFileManager`를 통해 요청별로 고유한 작업 디렉토리를 생성하고 실행 후 즉시 정리(Cleanup)하여, 동시 요청 간의 파일 충돌 및 스토리지 누수를 방지했습니다.

* **실시간 통신을 위한 경량화 메시지 브로커 도입**
    * **문제:** 채팅 기능을 구현하기 위해 DB(MySQL)를 주기적으로 조회(Polling)하는 방식은 실시간성이 떨어지고 데이터베이스에 불필요한 부하를 주는 비효율이 존재했습니다. 이를 해결하기 위해 메시지 브로커 도입이 필수적이라 판단했습니다.
    * **해결:** Kafka, RabbitMQ, Redis Pub/Sub을 비교 분석했습니다. 대규모 트래픽 처리가 필요한 상황이 아니었기에 Kafka 등의 무거운 솔루션은 오버 엔지니어링이라 판단하였고, **러닝 커브가 낮고 인메모리 기반으로 빠른 응답 속도를 제공하는 Redis Pub/Sub을 선택**하여 프로젝트 규모에 맞는 최적의 아키텍처를 구현했습니다.
    * **성과:** 복잡한 인프라 구축 없이 저지연 실시간 채팅 시스템을 완성했으며, 채팅 내역 저장은 MongoDB로 분리하여 RDBMS의 부하를 효과적으로 제거했습니다.

* **WebSocket 보안 프로토콜 강화**
    * **문제:** WebSocket 연결은 HTTP 핸드셰이크 이후 TCP 연결이 유지되므로, 기존 Spring Security의 HTTP 필터 체인만으로는 메시지 단위의 인증/인가를 수행하기 어려웠습니다.
    * **해결:** `ChannelInterceptor`를 구현하여 **STOMP 프로토콜 레벨의 보안**을 적용했습니다. `CONNECT` 프레임의 헤더에서 JWT를 추출하여 검증하고, 유효한 토큰이 아닐 경우 핸드셰이크 단계에서 연결을 차단하여 비인가 사용자의 접근을 원천 봉쇄했습니다.

여행 계획 플래너 및 커뮤니티 플랫폼입니다.

* **OAuth2 심층 학습 및 Stateless CSRF 방어 로직 설계**
    * **문제:** Google Cloud Run과 같은 서버리스 환경 특성상 **세션(`HttpSession`)을 유지하기 어려워**, 기존의 세션 기반 인증 방식을 사용할 수 없었습니다.
    * **해결:** 대안으로 **JWT**를 도입했으나, 표준 라이브러리인 `spring-boot-starter-oauth2-client`가 세션을 사용하는 Stateful 방식임을 파악했습니다. 이에 라이브러리를 사용하지 않고 **`RestTemplate`을 활용해 인증 흐름을 직접 구현**했습니다. 특히 CSRF 방지용 `state` 값까지 JWT로 생성하여 클라이언트에 발급하고 검증하는 **완전한 Stateless OAuth2 인증 로직**을 설계했습니다.
    * **성과:** 별도의 세션 저장소(Redis 등) 없이도 안전한 소셜 로그인을 구축하여 인프라 비용을 절감하고, OAuth2 프로토콜의 동작 원리를 깊이 있게 체득했습니다.

* **데이터 정합성 보장 및 고아 리소스 관리**
    * **문제:** 게시글 작성 중 이미지를 업로드했으나 최종 저장을 하지 않고 이탈할 경우 스토리지(GCS)에 '고아 객체'가 남는 문제가 있었습니다. 또한, 게시글 삭제 시 DB 트랜잭션은 롤백되었으나 이미지는 이미 삭제되어 버리는 정합성 문제가 우려되었습니다.
    * **해결:**
        1.  **임시 ID 시스템:** 게시글 작성 시 **음수 ID**를 임시 발급하여 이미지를 매핑하고, 최종 저장 시 실제 ID로 업데이트하는 로직을 구현했습니다.
        2.  **트랜잭션 동기화:** `TransactionSynchronizationManager`를 활용하여 DB 트랜잭션이 완전히 커밋된 직후(`afterCommit`)에만 GCS 이미지 삭제 요청을 보내도록 하여, 롤백 시 데이터 소실을 방지했습니다.
        3.  **자동 정리 스케줄러:** 매 정각마다 `NegativeImageCleanupJob`이 실행되어, 최종 저장되지 않고 남겨진 음수 ID의 이미지들을 GCS와 DB에서 일괄 정리하도록 자동화했습니다.

* **인프라 아키텍처 설계 및 클라우드 마이그레이션 (GCP → AWS)**

    * **문제:** 초기 구축 시 GCP Cloud Run과 Cloud SQL을 도입하여 인프라 관리 소요를 최소화했고, GitHub Actions를 연동하여 코드 푸시 즉시 배포되는 CI/CD 파이프라인을 운영했습니다. 하지만 GCP 프리티어 종료 후 운영 비용이 발생하기 시작했습니다. 
    * **해결:** AWS EC2와 RDS로 전체 인프라를 마이그레이션했습니다. 배포 환경 변화에 맞춰 Jenkins를 도입, Docker 이미지를 빌드하고 GHCR에 푸시한 뒤 EC2에서 실행하는 파이프라인을 재설계했습니다.
    * **성과:** PaaS(서버리스)와 IaaS(인스턴스) 환경을 모두 직접 구축하고 운영하며 각 아키텍처의 트레이드오프를 경험했습니다.
